# Trading Algorithm on the S&P 500

A project testing whether systematic trading signals, first using simple technical rules, then a
machine-learning pipeline, can generate returns on the S&P 500, built around rigorous
backtesting methodology: walk-forward validation, avoidance of look-ahead bias, and
cost-adjusted profit evaluation rather than stopping at classification accuracy.

## What's in here

- **`src/Trading_Algorithm.ipynb`** — the full notebook: rule-based baselines, feature
  engineering, an ML pipeline with walk-forward cross-validation, and a profit simulation with
  and without transaction costs.
- **`plots/`** — key charts generated by the notebook.

## Approach

1. **Rule-based baselines** — a 50/200-day moving-average crossover and a percentile-band
   mean-reversion strategy, backtested against buy-and-hold.

2. **Feature engineering** — standard technical indicators (SMA, EMA, momentum, RSI, MACD,
   Bollinger Bands, ATR) plus calendar features (weekday, cyclically-encoded day-of-year).

3. **ML pipeline** — several classifiers (logistic regression, random forest, gradient boosting,
   KNN, XGBoost, SVC) evaluated with `TimeSeriesSplit` walk-forward validation, so each model is
   trained only on data prior to the window it's tested on — avoiding a common source of
   inflated backtest results.

4. **Profit simulation** — predictions converted into long/short positions and compared to
   buy-and-hold, with and without a per-trade transaction cost assumption, to check whether
   classification performance actually translates into P&L.

## Key takeaways

- Transaction costs have a large effect on realised returns. A small per-trade cost assumption
  is enough to meaningfully change the picture versus a costless backtest.
- Classification accuracy alone doesn't guarantee profitability, which is why the notebook
  checks realised P&L directly rather than stopping at accuracy scores.
- Walk-forward validation was a deliberate design choice throughout, to avoid the model
  implicitly learning from future data.

## Possible next steps

- Test on less efficiently-priced instruments (small caps, less liquid assets).
- Incorporate higher-frequency or alternative data (order book, sentiment, macro releases)
  rather than price-only technical features.
- Model slippage and market impact more precisely than a flat per-trade cost assumption.

## Running it

Clone the repo, then install the dependencies and open the notebook:

```bash
pip install pandas numpy matplotlib seaborn yfinance pandas_market_calendars scikit-learn xgboost
jupyter notebook src/Trading_Algorithm.ipynb
```

No local data files are needed — the notebook pulls S&P 500 price history live via `yfinance`.
